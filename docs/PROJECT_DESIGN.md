# 国际贸易 ERP 全链路演示版 Demo 设计方案

## 0. 项目定位

### 0.1 新定位

本项目不再只是“扫码库存 Demo”，而是升级为：

```text
国际贸易 ERP 全链路成熟演示版 Demo
```

目标不是一次性做成完整企业级 ERP，而是让甲方看到一个接近成熟国际贸易 ERP 的完整系统形态。

本次升级后的核心原则是：

```text
核心链路真实可操作
+ 外围模块虚拟展示
```

也就是说：

- 核心业务闭环必须真实可跑通；
- 外围 ERP 模块允许使用固定数据、模拟状态流转、虚拟展示；
- 第一版重点是“像成熟 ERP”，而不是“把所有复杂业务都做深”。

### 0.2 必须保留并做深的真实核心闭环

以下能力必须真实实现，不能弱化：

1. 上传合同 / 箱单
2. AI 识别字段，第一版允许 mock
3. 生成合同和货物批次
4. 生成唯一二维码
5. 手机网页扫码入库
6. 手机网页扫码出库
7. 库存自动变化
8. 库存页真实统计
9. AI 助手回答真实库存数据

核心库存规则必须坚持：

```text
库存剩余数量必须由二维码状态真实计算，不能写死为 80。
```

### 0.3 第一版默认演示场景

默认演示主线仍然保留：

```text
中国采购 100 箱货
→ 发往赞比亚仓库
→ 上传合同和箱单
→ AI 识别信息
→ 生成合同和批次
→ 生成 100 个二维码
→ 手机扫码入库
→ 销售出库 20 箱
→ 库存剩余 80 箱
→ 老板问 AI：这批货现在还有多少？
```

但该场景只作为默认演示值，不允许写死在代码中。系统设计必须支持后续通过配置或表单调整：

- 商品名称
- 客户名称
- 供应商名称
- 目的仓库
- 总数量
- 单位
- 出库数量
- 合同金额
- 币种

### 0.4 第一版总体策略

第一版整体策略调整为：

- Demo 1.0：先把真实核心闭环打穿
- Demo 1.5：在真实闭环之上，包装成熟 ERP 的外围模块

## 1. 最重要开发原则

### 1.1 只做演示版，不做完整企业级 ERP

第一版不做：

- 完整 ERP 审批体系
- 复杂采购审批
- 真实物流接口
- 真实海关接口
- 真实报关规则引擎
- 完整成本分摊
- 复杂财务凭证
- 完整多公司权限隔离
- 多仓库复杂调拨
- 复杂 BI 报表
- 完整流程引擎
- 原生 App
- PDA 专用程序

第一版要做的是：

- 核心真实闭环
- ERP 形态完整展示
- 多模块联动的“可讲故事”演示系统

### 1.2 Codex 每次只做一个大环节

每次工作流程必须是：

```text
读取设计方案
→ 读取 TODO
→ 读取 PROJECT_MEMORY
→ 找到第一个未完成的大环节
→ 只实现这一个大环节
→ 自测
→ 更新 PROJECT_MEMORY
→ 更新 TODO
→ 停下来汇报并等待用户先验证
→ 用户确认满意
→ git commit
→ git push
```

没有用户确认：

- 不进入下一环节
- 不提前提交本地 git
- 不提前 push GitHub

### 1.3 核心真实、外围虚拟

必须遵守：

- 核心链路的数据和状态真实可操作
- 外围模块第一版允许使用 mock 数据、固定状态、模拟按钮
- 外围模块要体现它在完整 ERP 中的作用
- 外围模块要能和合同号、批次号、SKU、数量建立关联
- 外围模块要能体现“推进到下一状态”的流转感

### 1.4 AI 的边界

第一版允许：

- AI Mock 识别合同和箱单字段
- AI 模板化生成工单说明
- AI 根据真实库存统计结果生成自然语言回答
- AI 做单据一致性检查展示

第一版不允许：

- AI 直接改数据库
- AI 直接跳过用户确认写入业务数据
- 在前端直接暴露大模型 API Key

## 4. 各模块设计

### 4.1 首页驾驶舱

定位：

- 给甲方一个“接近成熟 ERP 首页”的第一视角
- 汇总核心真实数据和外围模块的演示状态

第一版展示：

- 合同数量
- 批次数量
- 在途批次
- 当前库存
- 已销售数量
- 未回款金额
- 待处理工单
- 最近单据
- 最近状态推进
- 订单池视角切换
- 主合同视角切换

数据来源原则：

- 库存、二维码数量必须来自真实二维码状态统计
- 成本、回款、在途等可以先使用演示数据或模拟状态
- 首页驾驶舱中的二维码指标默认展示系统内所有已生成二维码的全局累计，不跟随主合同视角切换
- 首页驾驶舱默认优先展示“进行中订单”，不把已完成订单长期堆在首页第一视角
- 首页驾驶舱需要支持订单池视角切换：
  - 进行中订单
  - 已完成待归档订单
  - 售后订单
  - 异常订单
  - 已归档订单
  - 全部订单
- 首页允许用户自主选择一个“主合同视角”，用于驱动右侧主信息区与执行进度区
- 首页中的订单状态不再只靠一个单字段表达，而是按四个维度展示：
  - 主流程状态
  - 售后状态
  - 异常状态
  - 归档状态
- 主合同切换后：
  - 全局库存总览仍保持展示全局真实统计
  - 主合同卡片、主批次、执行进度、主待办引用切换到所选合同
  - 如果所选合同尚未进入二维码/扫码环节，主视角显示该合同基础信息，但不伪造库存
- 如果当前订单池下暂时没有合同：
  - 首页允许保留全局库存总览与 ERP 模块总览
  - 但主订单视角必须诚实显示为空，不允许自动回退显示其他订单池的合同，避免误导甲方

订单状态体系原则：

- “订单完成”指主流程完成，不等于自动归档
- 全链路完成后，订单先进入“待归档”，由人工确认后再进入“已归档”
- 售后状态与主流程状态必须分离，不能因为触发售后就抹掉原有主流程进度
- 异常状态与主流程状态也必须分离，便于单独查看“异常但仍在推进”的订单
- 退货、退款、换货等售后动作不能直接删除原始库存与资金历史，正式版应采用冲销、逆向单据或补充记录方式处理

### 4.2 合同与单据

这是核心真实闭环入口。

第一版需要真实实现：

- 上传合同文件
- 上传箱单文件
- AI Mock 识别
- 展示识别字段
- 允许人工确认和修正字段
- 确认后生成合同和批次

同时要补齐单据生命周期治理：

- 草稿单据允许删除
- 已生成正式业务数据的单据禁止删除
- 已生成正式业务数据的单据只允许作废、替换或归档
- 删除、作废、替换都必须进入审计日志
- 单据状态变化不能反向影响库存

### 4.2.1 单据删除、作废与版本管理

第一版单据治理遵循以下规则：

1. 单据上传后，先进入 `Document` 草稿层。
2. `Document.extractedJson` 只是识别草稿，不直接等于正式业务数据。
3. 只有点击“确认生成业务数据”后，才会创建 `Contract`、`ContractItem`、`Batch`、`PurchaseOrder`、`Payment / Receivable`。
4. “确认生成业务数据”前，必须先满足同一票草稿业务已经具备 `合同 + 箱单` 两类已识别单据，配对口径为同一个 `contractNoDraft + batchNoDraft`。
5. 前端在条件未满足时，必须直接禁用“确认生成业务数据”按钮，并明确提示当前缺少哪类单据。
6. 后端 `POST /api/documents/:id/confirm` 也必须再次校验；即使绕过前端直接调接口，只要同票缺少 `合同` 或 `箱单`，仍然拒绝生成正式业务数据。
7. 未生成正式业务数据的单据允许删除。
8. 已生成正式业务数据的单据禁止物理删除，只能作废、替换或归档。
9. 删除、作废、替换都必须写入 `AuditLog`。
10. 删除、作废、替换都不能删除合同、批次、二维码、库存流水，也不能改变库存。
11. 库存始终只由 `QrItem.status` 与 `StockMovement` 决定，不因单据删除或作废而变化。

第一版建议的 `Document` 生命周期字段包括：

- `status`
- `isDeleted`
- `deletedAt`
- `deletedBy`
- `voidedAt`
- `voidedBy`
- `voidReason`
- `replacedByDocumentId`
- `relatedEntityType`
- `relatedEntityId`
- `businessCreated`
- `version`

第一版生命周期操作：

- 删除：仅针对未生成业务数据的草稿单据，默认采用软删除并隐藏出工作台。
- 作废：针对已生成业务数据的单据，只把 `Document.status` 更新为 `VOIDED`，保留原文件与业务数据。
- 替换：上传新版本单据，旧单据状态变为 `REPLACED`，新单据成为当前有效版本。
- 历史：可以查看同一单据链路下的版本演进。

### 4.2.2 Demo 2.0 合同与单据业务化深化

阶段 22 将“合同与单据”从基础上传识别，进一步升级为接近成熟 ERP 的合同资料治理入口。

本阶段新增原则：

- 合同必须具备类型分层，至少支持 `采购合同`、`销售合同`、`公司间协议`、`补充协议`。
- 合同页面可以展示计划数量、合同金额、执行进度、逾期风险、违约状态、收付款计划对比，但这些字段仍然不等于库存。
- 真实库存继续只由 `QrItem.status` 与 `StockMovement` 计算，不能因为合同执行进度或合同数量而伪造库存。
- 单据包完整性从“合同 + 箱单”扩展为成熟贸易资料包视角：合同、箱单、提单、发票、产地证、清关附件。
- 生成正式业务数据的最低门槛仍然是同票 `合同 + 箱单` 已识别；提单、发票、产地证、清关附件主要用于国际物流、清关、财务和合规资料完整性展示。
- 单据字段修正、AI 识别、替换、作废、归档、确认生成业务数据都需要形成可追溯留痕。
- 阶段 22 新增 `DocumentChangeLog`，用于补充 `AuditLog` 之外更贴近单据业务的版本链和差异留痕。

本阶段页面展示要求：

- `合同与单据` 页面继续作为上传、识别、人工修正、确认生成业务数据入口。
- 页面必须展示 6 类单据的资料包完整性，并提示哪些资料影响正式业务生成、哪些资料影响清关 / 财务。
- `合同数据` 页面展示人工确认后生成的正式合同层数据，包括合同类型、执行进度、收付款计划、逾期 / 违约风险、单据包完整性。
- 如果当前仍是空白演示起点，页面必须诚实显示为空，不允许自动伪造合同、批次、二维码或库存。

### 4.2.3 Excel 导入与 AI Mock 识别设计

合同与单据环节后续需要支持 Excel / CSV 作为结构化业务单据导入形式。

成熟业务系统中的定位原则：

- 盖章合同正文通常以 `PDF / Word / 图片 / 扫描件` 作为正式文件载体。
- Excel 通常用于结构化明细，不替代盖章合同正文。
- Excel 适合承载合同明细、箱单明细、发票明细、采购明细、批次明细、成本费用明细、回款计划等数据。
- Demo 版允许上传 Excel 后由 AI Mock 识别关键字段，但识别结果仍然只是草稿，不直接写入正式业务数据。

Excel 导入必须继续走现有安全链路：

```text
上传 Excel / CSV
→ 写入 Document 表
→ 后端解析 sheet / 表头 / 行数据
→ AI Mock 识别关键字段
→ 保存到 Document.extractedJson
→ 前端展示关键字段与明细行预览
→ 人工修正
→ 用户点击“确认生成业务数据”
→ 才写入 Contract / ContractItem / Batch / PurchaseOrder / Receivable
```

Excel 识别后的 `Document.extractedJson` 建议包含：

- `sourceFormat`：`excel` / `csv`
- `sheets`：原始 sheet 名称与行数摘要
- `headerMap`：系统识别到的表头映射
- `summaryFields`：合同号、批次号、客户、供应商、SKU、数量、单位、金额、币种、目的仓库等关键字段
- `lineItems`：Excel 明细行标准化结果
- `warnings`：缺失字段、重复行、数量不一致、表头无法识别等提醒

不同单据类型的第一版 AI Mock 字段建议：

- 合同 Excel：
  - 合同号
  - 客户
  - 供应商
  - 商品名称
  - SKU
  - 数量
  - 单位
  - 合同金额
  - 币种
  - 交期
  - 目的仓库
- 箱单 Excel：
  - 箱单号
  - 合同号
  - 批次号
  - SKU
  - 本批数量
  - 单位
  - 箱号范围
  - 目的仓库
  - 装箱日期
- 发票 Excel：
  - 发票号
  - 合同号
  - 金额
  - 币种
  - 数量
  - 单价
  - 付款条件

第一版实现边界：

- 优先支持标准模板 Excel / CSV。
- 第一版不做复杂自由格式 Excel 智能理解。
- 第一版不做多语言任意表头深度语义识别。
- 第一版不做复杂公式计算和跨 sheet 自动合并。
- 第一版可以只解析第一个有效 sheet，后续再升级多 sheet。
- Excel 导入不能直接增加库存。
- 库存仍然只能在生成二维码并扫码入库后，依据 `QrItem.status` 与 `StockMovement` 变化。

前端展示要求：

- 上传区域继续保留合同、箱单、提单、发票、产地证、清关附件、其他等单据类型。
- 文件选择器允许上传 `.xlsx`、`.xls`、`.csv`。
- 上传区域必须明确显示当前支持的文件格式，例如：
  - 合同正文：`.pdf`、`.doc`、`.docx`、`.png`、`.jpg`、`.jpeg`
  - 结构化明细：`.xlsx`、`.xls`、`.csv`
- 上传区域必须明确提示：不支持的文件格式将被拒绝上传。
- 前端在用户选择文件时就要先做一次格式校验：
  - 如果文件后缀或 MIME 类型不在允许范围内，必须立即提示“上传文件格式不支持，请上传 PDF、Word、图片或 Excel/CSV 文件”。
  - 不允许把错误格式文件继续送去 AI Mock 识别。
- 后端必须再做一次二次校验：
  - 如果前端被绕过，后端仍然要拒绝非法格式文件写入。
  - 接口返回值必须给出明确错误原因，不能只返回模糊失败。
- 识别结果区域需要同时展示：
  - 关键字段识别结果
  - Excel 明细表预览
  - 缺失字段和数量一致性提醒
- 人工修正仍然发生在草稿层。
- 用户确认生成正式业务数据前，页面必须提示“Excel 解析结果只是草稿，确认后才成为正式业务依据”。

后续成熟版升级方向：

- 支持用户配置字段映射模板。
- 支持多 sheet 合并识别。
- 支持 Excel 与 PDF / 图片合同互相校验。
- 支持合同数量、箱单数量、发票数量的一致性检查。
- 支持按 Excel 明细行生成多条 `ContractItem`、采购明细、成本明细或应收明细。

### 4.2.4 单据自动归票、同票识别与人工确认设计

随着合同、箱单、提单、发票、产地证、清关附件逐渐增多，系统不能再只靠用户自己记住“哪几份文件属于同一票业务”。

成熟版口径应升级为：

```text
单据上传
→ AI 识别字段
→ 系统自动做同票 / 同批次归组判断
→ 形成“单据资料包草稿”
→ 用户确认归组结果
→ 再确认生成正式业务数据
```

这里必须明确一条成熟 ERP 原则：

- 同一合同，不一定等于同一批次。
- 同一批次，也不一定只对应一份单据。
- 一个合同可以拆成多个实际到货 / 发货批次。
- 一个批次下可以挂合同、箱单、发票、提单、产地证、清关附件等多份资料。
- 系统必须允许：
  - `一个合同 -> 多个批次`
  - `一个批次 -> 多份单据`
  - `同一运输任务 -> 一个或多个批次`

第一版真实实现边界：

- 先实现“自动归票建议 + 人工确认归组”。
- 不直接做全自动不可逆归票。
- 不允许只靠文件名相似就自动生成正式业务关联。
- 不允许系统在低置信度时静默并错票。

#### 4.2.4.1 新增中间层：单据资料包草稿

系统需要新增一层中间对象，暂定为：

- `DocumentPackageDraft`
- `DocumentPackageItem`
- `DocumentMatchLog`

它们的作用不是直接替代正式业务数据，而是承接：

- 这几份单据是否属于同一票业务
- 这几份单据是否属于同一合同
- 这几份单据是否属于同一批次
- 当前归组是系统自动建议，还是人工确认锁定

核心原则：

- `Document` 仍然是原始单据草稿层。
- `DocumentPackageDraft` 是归票 / 归组层。
- `Contract / Batch / PurchaseOrder / Receivable` 才是正式业务数据层。

#### 4.2.4.2 归票判断优先级

系统第一版按以下优先级做自动归票：

1. 第一优先级：`contractNoDraft + batchNoDraft` 完全一致  
   - 直接视为同一票、同一批次的高置信匹配。

2. 第二优先级：`contractNoDraft` 一致，但 `batchNoDraft` 缺失  
   - 如果客户、供应商、商品、目的仓库、单位、数量等字段也基本一致，标记为：
   - `高置信候选`
   - 需要人工确认是否并入已有资料包，或新建一个该合同下的新批次资料包。

3. 第三优先级：提单、发票、产地证、清关附件  
   - 如果识别到了相同的合同号或批次号，则可自动建议挂到已有资料包。
   - 但仍保留人工改挂能力。

4. 冲突规则：
   - 合同号一致，但批次号不同：
     - 不能自动判成同一批次
     - 应提示“同合同下不同批次候选”
   - 批次号一致，但客户 / 供应商 / 目的仓库明显冲突：
     - 标记为“冲突待处理”
     - 禁止静默自动合并

5. 低置信规则：
   - 只因为文件名相似、金额接近、日期相近，不允许直接自动归入正式同票。
   - 最多只能作为“低置信候选提示”，等待人工决定。

#### 4.2.4.3 归票结果状态设计

第一版建议给单据增加归票状态口径，例如：

- `UNMATCHED`：未归组
- `AUTO_MATCHED`：系统自动高置信归组
- `NEEDS_CONFIRMATION`：系统给出候选，待人工确认
- `MANUAL_CONFIRMED`：人工确认归组
- `CONFLICTED`：存在冲突，禁止自动并入
- `IGNORED`：被人工标记为无关单据或暂不处理

页面展示时，不能只显示“已关联 / 未关联”，而是要明确告诉用户：

- 这是系统自动判断的
- 还是人工已经确认的
- 还是当前存在冲突

#### 4.2.4.4 前端页面展示要求

`合同与单据` 页面需要新增归票展示能力：

- 单据列表增加：
  - 归票状态
  - 当前资料包编号
  - 系统匹配置信度
  - 匹配原因摘要

- 单据详情区域增加：
  - 当前识别到的合同号、批次号
  - 系统建议归入哪个资料包
  - 候选资料包列表
  - 归票原因
  - 冲突提示

- 用户应能执行：
  - `确认归入该资料包`
  - `改归到其他资料包`
  - `新建资料包`
  - `标记为无关单据`
  - `取消人工锁定，重新按系统建议匹配`

#### 4.2.4.5 正式业务生成规则不变

即使实现了自动归票，以下底线仍然不变：

- 归票成功不等于库存变化。
- 归票成功不等于正式业务数据已经生成。
- 没有 `合同 + 箱单` 的已识别同票资料包，仍然不能点击“确认生成业务数据”。
- 正式业务生成时，系统应优先从“已人工确认或高置信自动归组的资料包”中取数据。

同时应区分三种业务情况：

1. `这是同一合同的新附件`
2. `这是同一合同下的新批次`
3. `这是完全另一票业务`

第一版不要求系统 100% 自动判断正确，但必须把这三种情况区分展示给用户，而不是混成一个“是否同一票”的二元结果。

#### 4.2.4.6 Demo 2.0 第一版最小可落地目标

本阶段的最小可落地目标是：

- 自动识别明显同票：`contractNoDraft + batchNoDraft` 一致。
- 对“合同号相同但批次号不同”给出清晰提示，不自动错并。
- 支持在页面上查看归票状态、候选结果和冲突说明。
- 支持人工确认归票、改挂资料包、新建资料包。
- 支持把提单 / 发票 / 附件逐步挂到同一资料包。

本阶段暂不做：

- 复杂多语言自由语义归票
- 完整机器学习打分模型
- 大批量历史单据自动归档重组
- 跨公司复杂权限下的归票隔离
- 完整运输批次与清关任务的多对多自动拆分

### 4.6.1 二维码身份设计原则

扫码相关设计必须先明确“货物码”和“业务动作”之间的关系：

1. 每一箱货物只允许贴一个唯一货物二维码。
2. 不允许为同一箱货分别生成“入库二维码”和“出库二维码”。
3. 货物二维码只代表货物身份，不直接代表入库动作或出库动作。
4. 当前扫描是入库还是出库，必须由当前页面、当前任务、当前工单、当前单据上下文共同决定。
5. 入库单、出库单、预收货单可以有自己的单据二维码，但这些单据二维码不能贴在货物上替代货物码。
6. 系统扫描到货物二维码后，必须先校验“当前操作上下文是否允许该货物执行本次动作”，再决定是否允许继续。

这条原则的目的不是追求复杂，而是防止仓库人员：

- 在错误页面扫对了货
- 在正确页面扫错了批次
- 把货物身份码误当作单据动作码
- 因重复扫码或跨任务扫码造成错误库存

### 4.6.2 扫码入库防呆规则

阶段 6 的扫码入库必须采用“先校验、后确认、再写库”的流程。

扫码后必须先完成以下校验：

1. 二维码是否存在。
2. 二维码是否属于当前合同。
3. 二维码是否属于当前批次。
4. 二维码是否属于当前预收货任务或当前入库任务。
5. 二维码当前状态是否为 `PENDING_INBOUND`。
6. 如果状态已经是 `IN_STOCK`，必须提示：`该货物已入库，不能重复入库`。
7. 如果状态已经是 `OUTBOUND`，必须提示：`该货物已出库，不能入库`。
8. 如果二维码属于其他批次，必须提示：`该货物不属于当前入库批次`。
9. 如果二维码属于其他仓库任务，必须提示：`该货物不属于当前仓库任务`。

在全部校验通过后，页面不能直接写入数据库，必须先展示确认信息：

```text
当前操作：扫码入库
二维码编号：
商品名称：
合同号：
批次号：
当前状态：
目标仓库：
是否确认执行该操作？
```

只有用户点击“确认入库”后，系统才允许：

- 更新 `QrItem.status = IN_STOCK`
- 更新 `QrItem.inboundAt = 当前时间`
- 更新 `QrItem.warehouse / warehouseId / locationId`
- 新增一条 `StockMovement` 入库流水

每次成功入库都必须写入完整流水，至少记录：

- 操作人
- 操作时间
- 仓库
- 合同
- 批次
- 二维码
- 状态变化
- 本次操作类型 = `INBOUND`

### 4.6.3 扫码出库防呆规则

阶段 7 的扫码出库同样必须采用“先校验、后确认、再写库”的流程。

扫码后必须先完成以下校验：

1. 二维码是否存在。
2. 二维码是否属于当前销售单、当前出库单或当前出库任务。
3. 二维码是否属于当前批次。
4. 二维码当前状态是否为 `IN_STOCK`。
5. 如果状态是 `PENDING_INBOUND`，必须提示：`该货物尚未入库，不能出库`。
6. 如果状态是 `OUTBOUND`，必须提示：`该货物已出库，不能重复出库`。
7. 如果状态是 `FROZEN`，必须提示：`该货物已冻结，不能出库`。
8. 如果状态是 `DAMAGED` 或 `LOST`，必须提示：`该货物状态异常，不能出库`。
9. 如果二维码不属于当前出库任务，必须提示：`该货物不属于当前出库单`。

在全部校验通过后，页面不能直接出库，必须先展示确认信息：

```text

二维码编号：

合同号：
批次号：
当前状态：
目标仓库：
是否确认执行该操作？
```

只有用户点击“确认出库”后，系统才允许：

- 更新 `QrItem.status = OUTBOUND`
- 更新 `QrItem.outboundAt = 当前时间`
- 新增一条 `StockMovement` 出库流水

每次成功出库都必须写入完整流水，至少记录：

- 操作人
- 操作时间
- 仓库
- 合同
- 批次
- 二维码
- 状态变化
- 本次操作类型 = `OUTBOUND`

### 4.6.4 扫码页面前端防呆设计

扫码页面必须始终把“当前操作上下文”放在最显眼位置，避免仓库人员进错页面后盲扫。

页面必须清楚显示：

1. 当前操作类型：入库 / 出库
2. 当前任务编号：预收货单号、入库单号或出库单号
3. 当前批次号
4. 当前合同号
5. 当前仓库
6. 应扫数量
7. 已扫数量
8. 剩余数量
9. 最近扫码记录
10. 异常扫码记录

入库页面和出库页面必须明显区分：

1. 页面标题必须不同。
2. 主按钮文案必须不同。
3. 确认弹窗文案必须不同。
4. 成功提示必须不同。
5. 错误提示必须醒目，并保留在异常扫码记录中。

页面交互必须遵守：

- 扫码成功不等于业务成功
- 只有“扫码命中 + 校验通过 + 用户确认”三者同时满足，才允许真正写库
- 异常扫码必须可追溯，便于现场复盘
- 最近扫码记录与异常扫码记录要分开展示，避免把错误记录混进成功记录中

### 4.14 AI 助手模块

这属于核心真实闭环模块。

第一版需要真实实现：

- 查询当前批次库存
- 查询某合同库存
- 查询今日入库
- 查询今日出库
- 查询某二维码生命周期
- 查询未回款合同（第一版可模板回答）
- 提供可直接演示的 `AI 助手` 问答页面

实现原则：

- AI 不能直接查数据库
- 后端先做受控查询
- AI 根据真实结果生成自然语言回答
- AI 不可用时使用模板回答
- 第一版采用 `规则识别意图 -> 后端查询真实数据 -> 大模型负责语言组织` 的结构
- 大模型只允许读取后端整理后的受控 JSON 结果，不允许自行拼接 SQL 或直接访问数据库
- 问答接口需要优先覆盖：
  - 批次库存
  - 合同库存
  - 指定日期 / 今日入库
  - 指定日期 / 今日出库
  - 单个二维码生命周期
  - 未回款合同摘要
- 当前项目必须继续支持：
  - 未配置大模型时，Demo 仍然可以用模板回答完整跑通
  - 已配置大模型时，只增强表述，不改变真实业务数据来源
- 大模型接入参数只允许放在后端环境变量，例如：
  - `AI_PROVIDER`
  - `AI_API_KEY`
  - `AI_MODEL`
  - `AI_BASE_URL`
- 阶段 9 增强版继续支持：
  - 默认使用“本地模板 AI”模式，开箱即可演示，不强制依赖外部大模型
  - 页面右下角 `AI 助手` 抽屉中允许切换到“升级版 AI 助手”
  - 升级版 AI 的 `provider / model / baseUrl / apiKey` 允许由网页表单输入
  - 网页输入的升级配置只允许保存在后端本地运行时配置中，不写入前端源码，也不写入业务数据库
  - 前端后续只能看到是否已配置与脱敏后的 Key 状态，不能回显完整 API Key
  - AI 运行时配置优先级为：网页保存的升级配置 > 后端 `.env` 配置 > 本地模板模式
  - 即使升级版 AI 调用失败，也必须继续回退到模板回答，保证 Demo 主链路不中断

## 9. Codex 工作规范

Codex 每次工作开始必须执行：

```text
1. 阅读 docs/PROJECT_DESIGN.md
2. 阅读 docs/TODO.md
3. 阅读 docs/PROJECT_MEMORY.md
4. 找到第一个未完成的大环节
5. 只实现这一个环节
6. 自测
7. 更新 PROJECT_MEMORY.md
8. 更新 TODO.md
9. 停下来汇报，等待用户先验证
10. 用户确认满意后 git commit
11. 用户确认满意后 git push
```

Codex 不允许：

- 一次性实现多个大环节
- 跳过 TODO 顺序
- 不更新记忆文件
- 在用户未确认前就提交 git
- 在用户未确认前就 push GitHub
- 直接把 API Key 写入前端
- 让 AI 直接修改数据库
- 未经用户确认进入下一阶段
