# 国际贸易 ERP 全链路演示版 Demo 设计方案

## 0. 项目定位

### 0.1 新定位

本项目不再只是“扫码库存 Demo”，而是升级为：

```text
国际贸易 ERP 全链路成熟演示版 Demo
```

目标不是一次性做成完整企业级 ERP，而是让甲方看到一个接近成熟国际贸易 ERP 的完整系统形态。

本次升级后的核心原则是：

```text
核心链路真实可操作
+ 外围模块虚拟展示
```

也就是说：

- 核心业务闭环必须真实可跑通；
- 外围 ERP 模块允许使用固定数据、模拟状态流转、虚拟展示；
- 第一版重点是“像成熟 ERP”，而不是“把所有复杂业务都做深”。

### 0.2 必须保留并做深的真实核心闭环

以下能力必须真实实现，不能弱化：

1. 上传合同 / 箱单
2. AI 识别字段，第一版允许 mock
3. 生成合同和货物批次
4. 生成唯一二维码
5. 手机网页扫码入库
6. 手机网页扫码出库
7. 库存自动变化
8. 库存页真实统计
9. AI 助手回答真实库存数据

核心库存规则必须坚持：

```text
库存剩余数量必须由二维码状态真实计算，不能写死为 80。
```

### 0.3 第一版默认演示场景

默认演示主线仍然保留：

```text
中国采购 100 箱货
→ 发往赞比亚仓库
→ 上传合同和箱单
→ AI 识别信息
→ 生成合同和批次
→ 生成 100 个二维码
→ 手机扫码入库
→ 销售出库 20 箱
→ 库存剩余 80 箱
→ 老板问 AI：这批货现在还有多少？
```

但该场景只作为默认演示值，不允许写死在代码中。系统设计必须支持后续通过配置或表单调整：

- 商品名称
- 客户名称
- 供应商名称
- 目的仓库
- 总数量
- 单位
- 出库数量
- 合同金额
- 币种

### 0.4 第一版总体策略

第一版整体策略调整为：

- Demo 1.0：先把真实核心闭环打穿
- Demo 1.5：在真实闭环之上，包装成熟 ERP 的外围模块

## 1. 最重要开发原则

### 1.1 只做演示版，不做完整企业级 ERP

第一版不做：

- 完整 ERP 审批体系
- 复杂采购审批
- 真实物流接口
- 真实海关接口
- 真实报关规则引擎
- 完整成本分摊
- 复杂财务凭证
- 完整多公司权限隔离
- 多仓库复杂调拨
- 复杂 BI 报表
- 完整流程引擎
- 原生 App
- PDA 专用程序

第一版要做的是：

- 核心真实闭环
- ERP 形态完整展示
- 多模块联动的“可讲故事”演示系统

### 1.2 Codex 每次只做一个大环节

每次工作流程必须是：

```text
读取设计方案
→ 读取 TODO
→ 读取 PROJECT_MEMORY
→ 找到第一个未完成的大环节
→ 只实现这一个大环节
→ 自测
→ 更新 PROJECT_MEMORY
→ 更新 TODO
→ 停下来汇报并等待用户先验证
→ 用户确认满意
→ git commit
→ git push
```

没有用户确认：

- 不进入下一环节
- 不提前提交本地 git
- 不提前 push GitHub

### 1.3 核心真实、外围虚拟

必须遵守：

- 核心链路的数据和状态真实可操作
- 外围模块第一版允许使用 mock 数据、固定状态、模拟按钮
- 外围模块要体现它在完整 ERP 中的作用
- 外围模块要能和合同号、批次号、SKU、数量建立关联
- 外围模块要能体现“推进到下一状态”的流转感

### 1.4 AI 的边界

第一版允许：

- AI Mock 识别合同和箱单字段
- AI 模板化生成工单说明
- AI 根据真实库存统计结果生成自然语言回答
- AI 做单据一致性检查展示

第一版不允许：

- AI 直接改数据库
- AI 直接跳过用户确认写入业务数据
- 在前端直接暴露大模型 API Key

### 4.2 合同与单据

这是核心真实闭环入口。

第一版需要真实实现：

- 上传合同文件
- 上传箱单文件
- AI Mock 识别
- 展示识别字段
- 允许人工确认和修正字段
- 确认后生成合同和批次

同时要补齐单据生命周期治理：

- 草稿单据允许删除
- 已生成正式业务数据的单据禁止删除
- 已生成正式业务数据的单据只允许作废、替换或归档
- 删除、作废、替换都必须进入审计日志
- 单据状态变化不能反向影响库存

### 4.2.1 单据删除、作废与版本管理

第一版单据治理遵循以下规则：

1. 单据上传后，先进入 `Document` 草稿层。
2. `Document.extractedJson` 只是识别草稿，不直接等于正式业务数据。
3. 只有点击“确认生成业务数据”后，才会创建 `Contract`、`ContractItem`、`Batch`、`PurchaseOrder`、`Payment / Receivable`。
4. 未生成正式业务数据的单据允许删除。
5. 已生成正式业务数据的单据禁止物理删除，只能作废、替换或归档。
6. 删除、作废、替换都必须写入 `AuditLog`。
7. 删除、作废、替换都不能删除合同、批次、二维码、库存流水，也不能改变库存。
8. 库存始终只由 `QrItem.status` 与 `StockMovement` 决定，不因单据删除或作废而变化。

第一版建议的 `Document` 生命周期字段包括：

- `status`
- `isDeleted`
- `deletedAt`
- `deletedBy`
- `voidedAt`
- `voidedBy`
- `voidReason`
- `replacedByDocumentId`
- `relatedEntityType`
- `relatedEntityId`
- `businessCreated`
- `version`

第一版生命周期操作：

- 删除：仅针对未生成业务数据的草稿单据，默认采用软删除并隐藏出工作台。
- 作废：针对已生成业务数据的单据，只把 `Document.status` 更新为 `VOIDED`，保留原文件与业务数据。
- 替换：上传新版本单据，旧单据状态变为 `REPLACED`，新单据成为当前有效版本。
- 历史：可以查看同一单据链路下的版本演进。

## 9. Codex 工作规范

Codex 每次工作开始必须执行：

```text
1. 阅读 docs/PROJECT_DESIGN.md
2. 阅读 docs/TODO.md
3. 阅读 docs/PROJECT_MEMORY.md
4. 找到第一个未完成的大环节
5. 只实现这一个环节
6. 自测
7. 更新 PROJECT_MEMORY.md
8. 更新 TODO.md
9. 停下来汇报，等待用户先验证
10. 用户确认满意后 git commit
11. 用户确认满意后 git push
```

Codex 不允许：

- 一次性实现多个大环节
- 跳过 TODO 顺序
- 不更新记忆文件
- 在用户未确认前就提交 git
- 在用户未确认前就 push GitHub
- 直接把 API Key 写入前端
- 让 AI 直接修改数据库
- 未经用户确认进入下一阶段
